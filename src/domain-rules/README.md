# Phase 3: Domain-Specific Rules (Re-Ranking)

> **Input:** Ranked candidates from Phase 2 + entity type + document context  
> **Output:** Re-ranked candidates with adjusted scores  
> **Paper reference:** Section 3, "Domain-Specific Rules", Table 2 (ablation)

## What does this do?

Takes the ranked candidates from Phase 2 and applies domain-specific rules to re-rank them. These rules use biomedical knowledge that pure string matching can't capture — like filtering out chemicals when looking for diseases, or boosting candidates whose definition matches the context.

**Ablation from BioLinkerAI paper:** Without domain rules = 87.3%, with = 93.3% (+6.0% on BC5CDR).

## Rules

### Rule 5: Multi-KG Confidence

Candidates found in multiple knowledge graphs (MeSH + Wikidata + DBpedia + UMLS) receive a confidence boost. The intuition: if an entity appears under the same identifier across independent knowledge bases, the mapping is more reliable.

```
Entity in 1 KG (MeSH only):     +0 boost
Entity in 2 KGs (MeSH + UMLS):  +5 boost
Entity in 3 KGs:                 +10 boost
Entity in 4 KGs:                 +15 boost
```

### Rule 6: Semantic Type Filtering

If the mention's entity type is known (e.g., "Disease" or "Chemical"), candidates with incompatible MeSH tree numbers are penalized. This prevents linking diseases to chemicals and vice versa.

```
MeSH tree number mapping:
  C* = Diseases              → compatible with "Disease"
  D* = Chemicals and Drugs   → compatible with "Chemical"
  F03 = Mental Disorders     → compatible with "Disease"
  B* = Organisms             → compatible with "OrganismTaxon"
```

Mismatched candidates receive a -30 penalty (configurable).

### Rule 7: Definition-Based Re-Ranking

Candidates whose MeSH definition (scope note) shares keywords with the mention or surrounding context get a score boost. This helps when string matching fails but the concept is semantically related.

```
Example: mention = "scleroderma renal crisis"
  Candidate "Kidney Diseases" has definition containing "renal" → boost
  Candidate "Scleroderma citrinum" (a fungus) has no relevant keywords → no boost
```

## Files

| File | What it does |
|---|---|
| `domain_rules.py` | `DomainRuleReranker` class — implements Rules 5-7 with configurable weights. |
| `evaluate.py` | Evaluation script: runs Phase 2 → Phase 3 on BC5CDR, compares Accuracy@1 before/after. |

## CLI usage

```bash
# Basic evaluation (Phase 2 + Phase 3)
python3 evaluate.py

# With all enrichment (needed for Rule 5 multi-KG)
python3 evaluate.py --wikidata --dbpedia --umls Data/UMLS/MRCONSO.RRF

# Tune rule weights
python3 evaluate.py --rule5-boost 5.0 --rule6-penalty -30.0 --rule7-boost 10.0

# Quick test with 200 mentions
python3 evaluate.py --wikidata --dbpedia --umls Data/UMLS/MRCONSO.RRF --limit 200
```

### Parameters

| Parameter | Default | Description |
|---|---|---|
| `--rule5-boost` | `5.0` | Score boost per additional KG source |
| `--rule6-penalty` | `-30.0` | Penalty for semantic type mismatch |
| `--rule7-boost` | `10.0` | Boost per keyword overlap in definition |
| `--top-k` | `10` | Number of Phase 2 candidates to re-rank |
| `--wikidata` | off | Load Wikidata cache for Rule 5 |
| `--dbpedia` | off | Load DBpedia cache for Rule 5 |
| `--umls` | off | Path to MRCONSO.RRF for Phase 2 enrichment |

## Quick usage (Python)

```python
from domain_rules import DomainRuleReranker

reranker = DomainRuleReranker(
    mesh_index=index,
    wikidata_synonyms=wikidata_cache,
    dbpedia_synonyms=dbpedia_cache,
    umls_synonyms=umls_cache,
)

reranked = reranker.rerank(
    mention="scleroderma renal crisis",
    candidates=candidates,        # from Phase 2
    entity_type="Disease",         # from annotation or Phase 1
    context="The patient developed scleroderma renal crisis...",
)
```

## Data dependencies

Same as Phase 2 — see `src/candidate-generation/README.md`. Rule 5 additionally needs the enrichment caches (generated by running Phase 2 with `--wikidata`, `--dbpedia`, `--umls`).
